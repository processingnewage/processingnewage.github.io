#### 进程理论

* **进程与线程**

  * 进程：进程就是程序在**内存**中的执行过程，是”活“的
  * 程序：程序就是存储在硬盘的代码，是“死”的
  
* **进程调度**
  
  * 先来先服务调度算法：对长作业有利，对短作业不利
  * 短作业优先调度算法：对短作业有利，对长作业无利
  * 时间片轮转法 + 多级反馈队列
  
* **进程运行的三状态图**

```text
（1）运行：当一个进程在处理机上运行时，则称该进程处于运行状态。处于此状态的进程的数目小于等于处理器的数目，对于单处理机系统，处于运行状态的进程只有一个。在没有其他进程可以执行时（如所有进程都在阻塞状态），通常会自动执行系统的空闲进程。

（2）就绪：当一个进程获得了除处理机以外的一切所需资源，一旦得到处理机即可运行，则称此进程处于就绪状态。就绪进程可以按多个优先级来划分队列。例如，当一个进程由于时间片用完而进入就绪状态时，排入低优先级队列；当进程由I／O操作完成而进入就绪状态时，排入高优先级队列。

（3）阻塞：也称为等待或睡眠状态，一个进程正在等待某一事件发生（例如请求I/O而等待I/O完成等）而暂时停止运行，这时即使把处理机分配给进程也无法运行，故称该进程处于阻塞状态。

  • 重要概念

    • 同步与异步

      描述的是任务的提交方式
	同步：任务提交过后，原地等待任务的返回结果，等待的过程中不做任何其他的事情，程序层面上表现出卡住的状态
	异步：任务提交后，不原地等待任务的返回结果，直接去做其他的事情，任务的返回结果会有一个异步回调机制自动处理

    • 阻塞与非阻塞

      描述程序的运行状态
	阻塞：阻塞态
	非阻塞：就绪态、运行态
理想状态：程序运行应该永远处于非阻塞态
      • 最高效的状态是异步非阻塞

开启进程

  • 代码开启进程和线程的方式一致

  • 在主函数中编写创建新的进程的方法，在运行之后，原代码是一个进程，在主函数中新创建一个进程

  • Windows操作系统下，创建进程一定要在main内，因为Windows下创建进程类似于模块导入的方式，会从上往下顺序执行代码，不在main函数中创建新的进程会导致死循环

  • Linux系统以复制代码的形式创建新的进程，不需要在main函数编写

  • 总结：

    • 创建进程就是在内存中申请一块空间将需要运行的代码放进去
    • 一个进程对应内存中一块独立的空间
    • 多个进程对应内存中多块独立的空间
    • 进程之间默认是无法直接交互的，如果想要交互需要借助第三方工具、模块

      join方法

  • join让主进程等待子进程代码运行结束后再继续运行，不影响其他子进程的执行

僵尸进程与孤儿进程

  • 僵尸进程：所有的进程都会步入僵尸进程，在开设的子进程之后，该进程死后不会立即释放占用的进程号，需要让父进程在一段时间之内仍然可以获取到该进程的一些基本信息

  • 孤儿进程：子进程存活，父进程意外死亡，OS会开设有相关机制专门管理孤儿进程回收相关资源

    • 行锁：表格的一行锁住，只能允许一个人同时操作

线程理论

  • 什么是线程

    • 进程：资源单位（起一个进程仅仅只是在内存空间中开辟一块独立的空间）
    • 线程：执行单位（真正被CPU执行的其实是进程里面的线程，线程指的就是程序的执行过程，执行程序中所需要的资源都找所在的线程中去找）
    • 将操作系统比喻成一个工厂，进程相当于车间，而线程就是车间里面的流水线
    • 每一个进程自带一个线程
    • 进程和线程都是虚拟单位，只是为了方便描述问题，是操作系统的抽象

  • 为什么需要线程

    • 开设进程：
      • 申请内存空间 耗费资源
      • ”拷贝代码“ 耗费资源

  • 开设线程

    • 一个进程内可以开设多个线程，在用一个进程内开设多个线程无需再次申请内存空间操作
    • 总结：
      • 开设线程的资源开销要远远小于进程的开销
      • 同一进程之间的数据是共享的

GIL全局解释器锁（Global Interpreter Lock）

  • GIL不是python的特点，而是CPython解释器的特点
  • 在Cpython解释器中，GIL是一把互斥锁，用来阻止同一个进程下的多个线程的同时执行（同一进程下的多个线程无法利用多核优势），因为Cpython解释器的内存管理不是线程安全的（内存管理：垃圾回收机制（引用计数、标记清除、分代回收））（垃圾回收是一个线程，与其他的线程同时运行时会出现数据错乱，所以全局解释器锁保证一个进程中同时只能有一个线程运行）
  • GIL保证解释器级别的数据安全
  • 针对不同的数据仍然要加不同的锁处理，GIL只能在解释器级别上做到隔离多个线程
  • ==解释型语言的通病：同一个进程下的多个线程无法利用多核优势==（解释执行的劣势）

GIL与普通互斥锁的区别

from threading import Thread, Lock
import time

money = 100
mutex = Lock()		# 创建锁对象

def task():
    global money
    mutex.acquire()		# 申请锁
    tmp = money			# 获取当前的数据，相当于查询
  	# time.sleep(1)		# 延时模拟IO操作
# 注释该代码，不做延时（或者I/O操作），则最后结果为0，可以体会GIL的作用
# 若程序中有延时（或者I/O操作）（GIL自动释放），则需要额外加锁，GIL不能达到锁的效果
    money = tmp - 1		# 相当于买票
    mutex.release()		# 释放锁

if __name__ == "__main__":
    t_list = []
    for i in range(100):
        t = Thread(target=task)
        t.start()
        t_list.append(t)
    for t in t_list:
        t.join()
    print(money)

# 100个线程启动之后，先去抢GIL，但进入延时或者IO后，GIL会自动释放，但正在执行的线程还有一把互斥锁，其他的线程虽然抢到了GIL但是抢不到互斥锁，最终GIL回到该线程处，继续处理数据

多进程与多线程的比较

  • 多进程和多线程有各自的优势，通常项目中多进程下面再开设多线程

  • 多线程是否有用要分具体情况：

    • 单核：4个任务（IO密集型/计算密集型）
    • 多核：4个任务（IO密集型/计算密集型）

  • 计算密集型 ，每个任务需要10s

    • 单核：
      • 多进程：额外的消耗资源
      • 多线程：节省开销
    • 多核：
      • 多进程：总耗时，10s+
      • 多线程：总耗时，40s+

  • IO密集型

  • 多核：

    • 多进程：相对浪费资源

    • 多线程：更加节省资源

死锁与递归锁

  • 死锁：程序阻塞（多个锁互相锁死）
  • 递归锁：可以被连续的acquire和release，但是只能被第一个抢到这把锁的程序执行上述操作，它的内部有一个计数器，每acquire一次计数加一，每release一次计数减一，只要计数不为0，那么其他人都无法抢到这把锁

  • 在并发编程中信号量指的是锁
